# Ya — Telux Identity Setup

How Telux identity provisioning works on Ya: first boot sequence, DID generation, Island creation, and key storage on MSM8937.

Reference: `ZAKO/PROTOCOLS/Telux-Protocol-v1.md` §4, `ZAKO/PACKAGES/zako-telux.md`

---

## What Happens at First Boot

When a user first boots Ya and completes the setup wizard:

1. **`telux-identd` initializes** — generates the Sovereign's Ed25519 keypair in the Keymaster 4.0 TEE (Qualcomm TrustZone on MSM8937). The private key never leaves the TEE.

2. **DID is derived** — the Sovereign's DID is computed from the public key:
   ```
   did:key:z6Mk<BASE58BTC(ed25519_public_key)>
   ```
   This DID is the Sovereign's permanent identity on Ya. It is self-issued with no external registry.

3. **Personal Island is created** — `telux-ledgerd` receives the CREATE record for the Personal Island, computes `island_id = BASE58URL(BLAKE3(sovereign_public_key || provisioning_epoch))`, and writes the genesis record to the ledger.

4. **OPEN records for static sub-entities** are written (System, Identity, Exchange, Journal, Health, Academy, People, Places — sub_entity 0–7).

5. **Provisioning ledger record** is written — a BIND record (task_code=0x0D) linking the DID to the device's provisioning epoch. This is the Sovereign's first record in their chain.

## Keymaster Configuration on MSM8937

Ya uses Keymaster 4.0 via the Qualcomm hardware service:

```ini
# In telux.conf:
key_backend           = hardware
keymaster_service     = android.hardware.keymaster@4.0-service-qti
```

Qualcomm's Keymaster 4.0 implementation stores private keys in QSEE (Qualcomm Secure Execution Environment — the TrustZone TEE). The keystore HAL enforces key binding to the device — keys cannot be migrated off the device even with root.

**Attestation:** Keymaster 4.0 supports key attestation (proving a key is hardware-backed). On Ya, attestation chains are stored as ATTEST records (task_code=0x0E) in the Identity sub-entity.

## Identity Lock Behavior

When the device screen locks, `telux-identd` enters identity-locked state:
- New signing operations are **queued, not executed**
- Incoming unsigned records are staged
- On unlock: staged queue is processed FIFO — each record signed, then posted to `telux-ledgerd`

This means no record enters the canonical chain unsigned. A PADS invoice created while the screen is locked is signed the moment the user unlocks — with the correct timestamp (wall_ts from record creation, not signing time).

## DID Document

The Sovereign's DID document is generated by `telux-identd` on demand. It declares:
- The Sovereign's public key (Ed25519)
- Service endpoints (carrier addresses — SMS, IP, QR)
- Supported Telux protocol version

For SMS transport, the service endpoint is the device's MSISDN (phone number). `telux-identd` reads the MSISDN from the SIM via the telephony stack when a SIM is present.

## Key Rotation

Key rotation replaces the signing key while maintaining identity continuity. Procedure:

1. New keypair generated in TEE
2. Old key signs a BIND record (task_code=0x0D) for the new key's DID
3. New key signs an ATTEST record (task_code=0x0E) asserting continuity with the old DID
4. Both records written as a compound group to the Identity sub-entity
5. `telux-identd` transitions to signing with the new key

The rotation chain is locally verifiable. Any counterparty holding the chain can verify that the new DID is the authentic continuation of the old one. No external authority is required.

```bash
# Initiate key rotation (Sovereign action; requires current key to be unlocked):
telux-ctl rotate-key
```

## Backup and Recovery

The Sovereign's private key is bound to this device's hardware. It cannot be exported. This means:

- **Device loss or damage:** the Sovereign's key is lost. They must establish a new identity on a replacement device.
- **Future mitigation (planned):** a key backup protocol using a trusted guardian Sovereign — the Sovereign's key material encrypted under the guardian's public key, transmitted via Telux exchange, recoverable by the guardian's device on Sovereign request. This is not implemented in v1.

For v1, document clearly in the user onboarding: **your identity is on this device. Back up your contacts and records regularly.**

---

## Multi-Island Setup

By default, Ya provisions one Personal Island at setup. Work Islands are created dynamically when the user creates a new work context in the PADS app.

A user who wants multiple identity contexts (e.g., personal identity + a business identity) can create a second Personal Island by repeating provisioning in a new user profile (Android multi-user). This is not the standard flow for Ya v1.

## Verifying Identity Setup

```bash
# Confirm identity is provisioned:
adb shell telux-ctl identity
# Expected output: DID, key fingerprint, provisioning date

# Confirm Personal Island exists:
adb shell telux-ctl islands
# Expected output: "Personal" island with sub-entities 0–7

# Confirm ledger has genesis records:
adb shell telux-ctl ledger tail 10
# Expected: BIND + CREATE + OPEN records from provisioning

# Confirm Keymaster key is hardware-backed:
adb shell keystore2_client attest --alias telux_sovereign_key
# Expected: attestation chain with securityLevel=TrustZone or StrongBox
```
