# Git-Informed Adoptions: Detailed Design for C + SQLite

## Five structural ideas from git's data model, implemented natively in ZAKO's existing stack

*Companion to `04-git-overlay-narrative.md` (conceptual mapping) and `05-git-efficiency-analysis.md` (cost analysis). This document expands each proposed adoption into enough detail for implementation planning.*

---

## 1. Pack-Like Record Compaction

### The problem it solves

The `records` table in `ledgerd_store` grows linearly with every transaction. Each row stores the full frame blob (up to 64 bytes), the 32-byte frame_hash, the 32-byte chain_hash, plus integer columns — roughly 140-160 bytes per record after SQLite overhead. Over a year of moderate use (50 records/day), that is ~2.7MB. Over five years with heavier use, the database approaches tens of megabytes. On a 16GB eMMC where the OS, apps, and user data all compete for space, and where eMMC write endurance is finite, unbounded growth of the ledger is a long-term sustainability problem. The current design has no mechanism for reclaiming space or reducing the on-disk footprint of historical records that are rarely accessed.

Git solves the equivalent problem with pack files. Loose objects accumulate during active work, then `git gc` sweeps them into a single pack file with a companion index. Inside the pack, objects are delta-compressed against each other — if two commits differ by only a few bytes, the second is stored as a delta referencing the first. A repository with 100,000 objects might have a pack file 10x smaller than the sum of the loose objects. The pack index enables O(1) lookup by hash without scanning the entire file. ZAKO's records are ideal candidates for delta compression because BitLedger frames within the same batch share the same Layer 2 header (6 bytes) and often have similar flag patterns — only the value block and account pair vary.

The implementation adds a `packs` table: `packs(id INTEGER PRIMARY KEY, start_seq INTEGER, end_seq INTEGER, record_count INTEGER, pack_blob BLOB, index_blob BLOB, created INTEGER)`. A compaction job — triggered during FULL power mode when CPU and battery are abundant — reads a range of sequential records, groups them into a pack, delta-compresses them using a lightweight algorithm (copy/insert instruction pairs, similar to git's delta format but without the full xdelta machinery), writes the result as a single BLOB, and builds a small index (array of `[seq, offset]` pairs) for random access within the pack. The original `records` rows have their `frame` column set to NULL and a `packed INTEGER` flag set to 1, indicating that the raw frame should be retrieved from the pack. The frame_hash and chain_hash columns remain — they are needed for chain verification and are not compressible (hashes are pseudorandom). This means chain verification continues to work without touching pack data at all. Only frame *retrieval* (for display, export, or retransmission) goes through the pack. The compaction job never runs during CONSERVE or CRITICAL modes. It never modifies chain_hash or frame_hash. It never deletes records — only NULLs the frame blob. A future `VACUUM` pass can reclaim the freed pages within SQLite.

## 2. Bundle-Based Sync for sharedb

### The problem it solves

The current `telux-sharedb` architecture transmits records one at a time. `sdb_enqueue()` accepts a single frame, selects a carrier, formats it (pads-v1 URL for SMS, raw bytes for IP/BLE), and sends it. The counterparty receives, validates, and posts to their ledger. This works, but it means each record is an independent transmission with no context: the receiver cannot verify that the record belongs to a valid chain without already having the sender's chain state. If records arrive out of order (common with SMS), the receiver cannot reconstruct the chain. If the receiver is new (first exchange with this counterparty), they have no chain context at all. The receiver must trust the individual record's signature without being able to verify that it is consistent with the sender's ledger history.

Git's bundle format solves this by packaging a segment of history as a self-contained, verifiable unit. A bundle includes the objects (commits, blobs) and the ref pointers (branch tips) for a range of history. The receiver can verify the entire bundle's chain integrity on receipt, even without any prior state from the sender. The bundle is the minimum viable proof: "here is what happened, here is the chain that proves it, here is the signature that authenticates it." For ZAKO, a record bundle would contain: a sequence of BitLedger frames, their chain hashes, the chain tip hash, and the sender's signature over the bundle. The receiver verifies by recomputing the chain from the bundle's start point (either genesis, or a common ancestor both parties agree on) forward through all included records, checking that the final chain_hash matches the signed tip.

The concrete format is a binary structure: `[magic "ZBN1" (4 bytes)][version (1)][flags (1)][record_count (2)][common_ancestor_hash (32) or zeros if from genesis][records: for each: [frame_len (2)][frame (N)][chain_hash (32)]][tip_chain_hash (32)][sender_did_len (1)][sender_did (N)][signature (64)]`. Total overhead beyond the raw frames: ~135 bytes fixed + 34 bytes per record (chain_hash + length prefix). For a 10-record bundle of 5-byte frames: 135 + (10 × 39) = 525 bytes. Base64url: ~700 characters — too large for a single SMS, but splittable across 2-3 SMS segments using continuation frames (the C0 Continuation flag exists for exactly this purpose). For IP and BLE, the bundle transmits as raw binary with no encoding overhead. The receiver's `sharedb` processes the bundle atomically: verify chain, verify signature, then `lds_append()` each record in sequence. If any chain_hash fails verification, the entire bundle is rejected — no partial state contamination. This is strictly stronger than the current per-frame approach, where a corrupted frame is rejected but the receiver has no way to know whether previous individually-accepted frames were part of the same valid chain.

## 3. Branch-Based Island Model in SQLite

### The problem it solves

The current `ledgerd_store` maintains a single chain. `store->last_seq` and `store->last_chain_hash` track one chain tip. The Islands concept — personal, work, and potentially others — is expressed as a directory structure (`/data/zako/islands/personal/`, `/data/zako/islands/work/`) but is not reflected in the ledger's data model. If both Islands write records to the same `records` table, they share one chain. A record in the personal Island's chain_hash depends on the previous record, which might be a work Island record. This means the two Islands are not independently verifiable — verifying the personal Island requires reading work Island records too, because the chain weaves through both. It also means a corruption in one Island's records breaks the chain for the other. The Islands are conceptually independent but structurally entangled.

Git's branch model provides the answer: each branch is an independent commit chain with its own root (genesis) and its own tip. Branches never share chain state unless explicitly merged. In SQLite terms, this means adding a `chains` table that tracks each independent chain: `chains(id INTEGER PRIMARY KEY, name TEXT UNIQUE NOT NULL, genesis_hash BLOB, tip_hash BLOB NOT NULL, tip_seq INTEGER NOT NULL, created INTEGER NOT NULL)`. The `records` table gains a `chain_id INTEGER NOT NULL` column with a foreign key to `chains`. Each Island is a row in `chains`. The `lds_store_t` struct changes from holding a single `last_seq` / `last_chain_hash` to holding a `current_chain_id` that is set by the caller before appending. `lds_append()` reads the tip of the specified chain (from the `chains` table or a per-chain in-memory cache), computes chain_hash against that chain's tip, appends the record with the chain_id, and updates the chain's tip. Chain verification becomes scoped: `lds_verify_chain(store, chain_id, seq_start, seq_end)` only walks records belonging to that chain.

This separation has three immediate consequences. First, Islands become independently auditable — verifying the personal ledger touches zero work records, and vice versa. Second, new Island creation is a lightweight operation: `INSERT INTO chains (name, genesis_hash, tip_hash, tip_seq, created) VALUES ('work', zeros_32, zeros_32, 0, now)` — no directory creation, no filesystem operation, just a row. Third, the power ledger (`outstack-powerd`'s MODE_CHANGE records) and the capability ledger (`identd`'s GRANT/REVOKE records) can each be their own chain, queryable and verifiable independently. The `chains` table becomes the registry of all ledger namespaces on the device, analogous to git's `refs/` directory. A query like "list all Islands and their current state" becomes `SELECT name, tip_seq FROM chains` — one query, no filesystem traversal. The foreign key constraint ensures that every record belongs to exactly one chain, preventing the entanglement problem. The per-chain in-memory cache (an array of `{chain_id, last_seq, last_chain_hash}` structs, sized to `MAX_CHAINS`) avoids a SQLite read on every append — the chain tip is always in memory for active chains, loaded from the `chains` table at daemon startup.

## 4. Signed Record Commits

### The problem it solves

In the current architecture, ed25519 signing happens at the exchange layer: `exchange_engine.c` requires signatures on each leg of a bilateral exchange, and `identd_daemon` provides signing as a service. But the ledger store itself — `ledgerd_store.c` — is signature-unaware. When `lds_append()` writes a record, it computes frame_hash and chain_hash but does not sign anything. The chain_hash provides tamper evidence (if someone modifies a stored record, the chain breaks), but it does not provide *attribution* — the chain_hash doesn't prove *who* wrote the record, only that the records haven't been reordered or altered since they were written. For internal records (power mode changes, routine transactions within one device), attribution isn't critical — the device is the only writer. But for records that may need to be presented as evidence to a counterparty or auditor (exchange records, capability grants, field signoffs in PADS), the chain_hash alone is insufficient. The counterparty needs to verify not just that the chain is intact but that a specific identity authored a specific record.

Git's signed commits provide exactly this: an ed25519 (or GPG) signature over the commit hash, embedded in the commit object. Anyone with the author's public key can verify that the commit was authored by that key. The signature covers the commit hash, which covers the tree hash and parent hash, which transitively covers the entire history. Signing one commit effectively signs the entire chain up to that point. This is a powerful property that ZAKO can adopt selectively: not every record needs a signature (signing is expensive — ~20,000 instructions for ed25519 on ARM32), but records at critical junctures do. An exchange settlement record should be signed. A capability GRANT should be signed. A PADS field signoff should be signed. A routine power mode transition or a batch-internal transaction does not need a signature — the chain_hash already guarantees its integrity within the local device's trust boundary.

The implementation adds a `signature BLOB` column to the `records` table (NULLable — most records will have no signature). `lds_append()` gains an optional `const uint8_t *sig` parameter (NULL for unsigned records). When present, the signature is computed over the chain_hash: `ed25519_sign(chain_hash, seckey)`. This means the signature covers not just the current record but the entire chain state at the point of signing — identical to how git's signed commit covers the full ancestry. Verification is: extract the public key from the signer's DID (`zako_did_to_pubkey`), recompute the chain_hash for the record in question, and `ed25519_verify(chain_hash, sig, pubkey)`. The `lds_verify_chain()` function gains an optional mode where it also checks signatures on signed records, reporting both chain integrity and signature validity. The selective approach means the system pays the ed25519 cost only when bilateral proof is needed. A device that processes 50 internal records per day and 5 signed exchange records per day pays ~100,000 instructions for signing (5 × 20,000) rather than ~1,000,000 (50 × 20,000) — a 10x reduction compared to signing everything. The daemon-side implementation is clean: `ldd_process_frame()` in `ledgerd_daemon.c` already knows whether a frame came from an exchange (via the bus category subscription). Exchange-origin frames get signed before `lds_append()`; internal frames do not.

## 5. Merge-Like Bilateral Exchange Records

### The problem it solves

The current exchange engine (`exchange_engine.c`) manages bilateral exchanges through a state machine: PENDING_OUTBOUND, PENDING_INBOUND, COMPLETED. When an exchange completes, `exe_acknowledge()` or `exe_complete_outbound()` calls `post_atomic()`, which posts both legs (our frame and the counterparty's frame) to the ledger. But these two frames are posted as two separate records in the `records` table. Their bilateral relationship is implicit — they share the same exchange_id in the exchange engine's in-memory state, but once posted to the ledger, they are just two adjacent records with no structural link. If someone queries the ledger and finds a SEND record at seq 47, there is no direct way to find the corresponding RECEIVE record at seq 48 without knowing that they were part of the same exchange. The exchange engine's in-memory `exe_exchange_t` struct is transient — it doesn't survive a daemon restart. After posting, the bilateral relationship exists only as a convention (adjacent records with complementary direction flags), not as a structural guarantee.

Git's merge commit solves this definitively. A merge commit has two (or more) parent hashes. The commit itself represents the point where two independent histories converged. Both parents are explicit, stored in the commit object, and part of the hash computation. Anyone examining the merge commit can follow both parent pointers to see the two histories that were combined. The merge commit's hash covers both parents, so it is cryptographic proof that both histories existed at the time of the merge. For ZAKO, the bilateral exchange settlement is precisely a merge: our chain and the counterparty's chain converge in a single record that proves both legs existed and conserved.

The implementation adds a `parent2_chain_hash BLOB` column to the `records` table (NULLable — only non-NULL for settlement records). Normal records have one parent: `chain_hash = BLAKE3(frame_hash || prev_chain_hash)`. Settlement records have two parents: `chain_hash = BLAKE3(frame_hash || our_prev_chain_hash || counterparty_chain_hash)`. This three-input hash means the settlement record's chain_hash is cryptographic proof that both chain states existed at settlement time. The `post_atomic` callback in the exchange engine changes: instead of posting two separate records, it posts one settlement record that embeds both frames (concatenated in the frame blob, with a separator or length prefix) and carries both parent chain_hashes. The `parent2_chain_hash` column stores the counterparty's chain tip at the time of exchange. Verification is: given a settlement record, extract both parent hashes, verify that `parent1` matches the preceding record on our chain, verify that `parent2` is a valid chain tip signed by the counterparty (using the Signed Record Commits mechanism from adoption 4), and verify that the frame blob contains two legs that conserve. This transforms the bilateral exchange from an application-level convention into a ledger-level structural guarantee. A counterparty disputing a transaction can be shown the settlement record: it contains the counterparty's own chain_hash as parent2, and the settlement's chain_hash was computed over both parents — proving the exchange happened at that exact point in both chains' histories. The record is self-proving in the same way a git merge commit is self-proving.

## 6. Prepared Statement Caching

### The problem it solves

Every call to `lds_append()` invokes `sqlite3_prepare_v2()` to compile the INSERT SQL string into bytecode. This compilation involves tokenizing the SQL text (scanning ~200 characters), parsing tokens into an AST, running the query planner (which examines table schema and indexes), and generating SQLite's VDBE bytecode. On the Cortex-A53, this costs ~2,500 instructions — nearly 30% of the total CPU work per record. The SQL string never changes between calls. The schema never changes at runtime. The query plan is always the same: insert into the `records` table's clustered index with two secondary index updates. Every invocation pays the full compilation cost for an identical result.

The fix is straightforward: prepare the statement once during `lds_open()`, store the compiled statement handle in the `lds_store_t` struct, and reuse it across all `lds_append()` calls. Between calls, `sqlite3_reset()` clears the statement's bindings and state (cost: ~20 instructions) without discarding the compiled bytecode. The same applies to the batch UPDATE: currently constructed via `snprintf()` into a fresh SQL string each time, it should be a prepared `UPDATE batches SET record_count=record_count+1, balance=? WHERE id=?` with bind parameters. The `lds_store_t` struct gains two new fields: `sqlite3_stmt *insert_stmt` and `sqlite3_stmt *batch_update_stmt`, both prepared during `lds_open()` and finalized during `lds_close()`. The combined savings are ~5,300 instructions per record — the INSERT preparation drops from ~2,500 to ~20 (just `sqlite3_reset`), and the batch UPDATE drops from ~3,200 (snprintf + parse + plan + execute) to ~220 (reset + bind + step). This is the single highest-value optimization available in the current codebase, reducing the total per-record CPU cost from ~8,750 to ~3,450 instructions. It requires changing roughly 30 lines of code in `ledgerd_store.c`.

## 7. Cursor-Based Chain Verification

### The problem it solves

`lds_verify_chain()` verifies chain integrity by iterating from `seq_start` to `seq_end`, calling `lds_get_by_seq()` for each record. Each call to `lds_get_by_seq()` prepares a new SQL statement (`SELECT ... WHERE seq=?`), binds the sequence number, steps once, copies the result into an `lds_record_t`, and finalizes the statement. For a 1,000-record verification, this means 1,000 separate SQL compilations, 1,000 B-tree lookups (each starting from the root page), and 1,000 statement finalizations. The B-tree lookups are particularly wasteful: records are stored in sequential order by `seq` (it's the primary key), so the B-tree pages for seq 47 and seq 48 are almost certainly on the same leaf page — but each lookup starts from the root and walks down, doing 3-4 page reads each time.

A cursor-based approach uses a single `SELECT seq, frame, frame_hash, chain_hash FROM records WHERE seq BETWEEN ? AND ? ORDER BY seq` query, prepared once, that returns records sequentially. SQLite's B-tree cursor walks leaf pages left-to-right, reading each page exactly once. For 1,000 sequential records occupying ~50 leaf pages, this is 50 page reads instead of 3,000-4,000 (1,000 lookups × 3-4 pages per lookup). The query is prepared once (~2,500 instructions), stepped 1,000 times (~200 instructions per step for row decoding), and finalized once (~100 instructions). Total: ~202,600 instructions vs. the current ~2,700,000 (1,000 × prepare + bind + step + finalize ≈ 2,700 each). A 13x reduction in CPU work for chain verification. The I/O reduction is even more dramatic: 50 sequential page reads vs. ~3,500 random page reads. On eMMC, sequential reads are 5-10x faster than random reads because the flash translation layer can prefetch adjacent pages. A 1,000-record chain verification that currently takes ~200ms would complete in ~15-20ms. The implementation replaces the `for` loop in `lds_verify_chain()` with a single prepared statement and a `while (sqlite3_step() == SQLITE_ROW)` loop that recomputes chain_hash incrementally, carrying the previous record's chain_hash in a local variable rather than re-fetching it.

## 8. Manual Bitshift Extraction for Hot Fields

### The problem it solves

The `extract_bits()` function in `zako_bitledger.c` is a generic bit-field extractor that loops bit-by-bit. For each bit, it computes the byte index (`bit_pos / 8`), the bit index within the byte (`7 - (bit_pos % 8)`), extracts the bit, and shifts it into the result. The division and modulo operations each cost 4-8 cycles on the Cortex-A53 (which has no hardware divide — it uses iterative algorithms). For the 25-bit `value_n` field, this means 25 iterations × ~8 instructions per iteration = ~200 instructions. For the full Layer 3 decode (`zbl_record_decode`), the function is called 13 times for fields totaling 40 bits: ~320 instructions of bit extraction. The function is correct, portable, and readable — it handles arbitrary start positions and widths cleanly. But it is called on every record decode, which happens on every frame received by `telux-ledgerd`, every frame displayed by the PADS app, and every frame verified during chain checks.

The hot fields have fixed positions known at compile time. The 25-bit `value_n` occupies bits 1-25 of a 5-byte array, which is bytes 0-3 with specific bit alignments. A manual extraction using shifts and masks computes the same result in ~15 instructions: load 4 bytes, shift each into position, OR them together, mask to 25 bits. No division, no modulo, no loop overhead. The same technique applies to account_pair (4 bits from bits 33-36: one byte load, one shift, one mask = ~5 instructions vs. ~32), direction (1 bit from bit 29: one byte load, one shift, one AND = ~3 instructions vs. ~8), and the other flag fields. The `extract_bits()` function should be retained for non-hot-path decoding (Layer 2 decode, which happens once per batch rather than once per record), but the Layer 3 decode should use direct bitshift operations. The same applies to `set_bits()` in `zbl_record_encode()`. The total savings per Layer 3 decode are ~250 instructions. For a chain verification of 1,000 records (each decoded during verification), this is 250,000 instructions saved — roughly equivalent to one fdatasync in CPU time, but entirely in userspace with no kernel transitions.

## Summary of Combined Impact

| Adoption | Type | Per-Record Savings | Implementation Effort |
|---|---|---|---|
| Pack-like compaction | New capability | Indirect (disk space, eMMC wear) | 2-3 days |
| Bundle-based sync | New capability | Indirect (fewer transmissions, stronger proof) | 3-4 days |
| Branch-based Islands | Schema change | ~10 instructions (one bind), major architectural clarity | 1-2 days |
| Signed record commits | Schema change | 0 for unsigned; ~20,000 cost for signed (selective) | 1 day |
| Merge-like bilateral records | Schema change | 0 for normal; ~900 for settlement (extra hash input) | 1 day |
| Prepared statement caching | Code fix | ~5,300 instructions saved | 2 hours |
| Cursor-based verification | Code fix | 13x faster chain verification | 2 hours |
| Manual bitshift extraction | Code fix | ~250 instructions per decode | 2 hours |

The three code fixes alone reduce the per-record append cost from ~8,750 to ~3,450 instructions. The five architectural adoptions add git's structural strengths (independent chains, bundled sync, signed records, merge-like settlements, compaction) without any of git's runtime costs (loose objects, zlib, text formatting, SHA-256, libgit2 dependency).

---

*Each adoption is independent and can be implemented in any order. The recommended sequence is: prepared statement caching first (highest impact, lowest effort), then branch-based Islands (enables all per-chain features), then signed record commits + merge-like settlements (depend on Islands), then bundle-based sync (depends on signed commits for bundle verification), then pack-like compaction last (a maintenance optimization, not a functional requirement).*
